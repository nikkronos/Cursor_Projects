# Техническое задание: Настройка GitHub Actions для автоматических тестов

## Что надо сделать
Настроить GitHub Actions workflow для автоматического запуска тестов при каждом push в репозиторий. При падении тестов система должна отправлять уведомления разработчику.

## Как это должно работать
- При каждом `push` в любую ветку репозитория автоматически запускается workflow
- Workflow устанавливает Python, зависимости из `requirements.txt`
- Запускается `pytest` для всех тестов в папке `tests/`
- Если тесты не проходят, GitHub отправляет уведомление (через email или в интерфейсе GitHub)
- Результаты тестов отображаются в разделе "Actions" репозитория
- Статус тестов отображается в виде badge/индикатора в README (опционально)

## Какие инструменты нам понадобятся
- GitHub Actions (встроенный CI/CD сервис GitHub)
- Python (версия из requirements.txt или последняя стабильная)
- pytest (уже в requirements.txt)
- Стандартные действия GitHub Actions (actions/checkout, actions/setup-python)

## Потенциальные проблемы и риски
- **Риск 1:** Тесты могут требовать переменные окружения или доступ к БД
  - **Митигация:** Проверить, что тесты не требуют реальных токенов или БД. Если требуют - использовать mock или тестовые данные через GitHub Secrets
- **Риск 2:** Разные версии Python на локальной машине и в GitHub Actions
  - **Митигация:** Явно указать версию Python в workflow (например, 3.9 или 3.10)
- **Риск 3:** Зависимости могут не установиться из-за проблем с сетью или версиями
  - **Митигация:** Использовать кеширование pip, указать точные версии в requirements.txt
- **Риск 4:** Тесты могут быть медленными и замедлять процесс разработки
  - **Митигация:** GitHub Actions выполняются асинхронно, не блокируют локальную работу. Можно настроить запуск только для определенных веток

## Альтернативные решения
- **Вариант А:** Использовать другие CI/CD сервисы (Travis CI, CircleCI, GitLab CI)
  - Минусы: Требуют дополнительной настройки, могут быть платными, менее интегрированы с GitHub
- **Вариант Б:** Запускать тесты вручную перед каждым коммитом
  - Минусы: Легко забыть, не автоматизировано, не работает для pull requests
- **Вариант В:** Использовать pre-commit hooks для запуска тестов локально
  - Минусы: Требует настройки на каждой машине разработчика, не работает для внешних PR
- **Выбран:** GitHub Actions (почему: бесплатно для публичных репозиториев, встроенная интеграция с GitHub, легко настроить, автоматические уведомления)
