# Правила работы с проектом TradeTherapyBot

## Язык общения

- **Пользователь общается на русском языке** и базово понимает английский.
- **AI‑агент может вести внутреннюю работу на английском** (комментарии в коде, названия переменных, часть документации), если это помогает экономить контекст.
- **Коммуникация с пользователем** всегда на русском языке, с максимально понятными объяснениями.

## Контекст и репозитории

- Этот файл описывает правила **конкретно для проекта TradeTherapyBot**.
- Общие правила для всех проектов и общей структуры лежат в `Main_docs/RULES_CURSOR.md` в репозитории `nikkronos/Cursor_Projectcs` (монорепо на GitHub).

### Серверы

- Боевой бот развёрнут на **сервере Timeweb**.
- Все секреты (токен бота, строки подключения к БД и т.п.) хранятся:
  - локально в `env_vars.txt` или `.env`;
  - на сервере — в отдельном конфигурационном файле.
- **Секреты никогда не коммитятся** ни в этот репозиторий, ни в другие.

Подробнее про сервер: см. `docs/server-timeweb.md` (в `Cursor_Projects`) и документацию в этом репозитории.

### GitHub‑репозитории

- Основной репозиторий кода бота: **`nikkronos/TradeTherapyBot`**.
- В репозитории **`nikkronos/Cursor_Projectcs`** может храниться копия структуры проектов и общая документация, но он **не является источником правды для кода бота**.
- Все изменения логики TradeTherapyBot должны в первую очередь попадать в репозиторий `TradeTherapyBot`.

## Структура документов в проекте

В корне проекта TradeTherapyBot (этого репозитория или папки `Projects/TradeTherapyBot/` внутри `Cursor_Projects`) должны быть:

### 1. `ROADMAP_TRADETHERAPYBOT.md`

**Назначение:** планы и приоритеты развития бота.

- Список задач с приоритетами.
- Отложенные идеи.
- Общий вектор развития.

### 2. `DONE_LIST_TRADETHERAPYBOT.md`

**Назначение:** история выполненных задач.

- Все важные задачи, выполненные по проекту.
- Дата, краткое описание результата.
- Можно ссылаться на PR/коммиты.

### 3. `SESSION_SUMMARY_YYYY-MM-DD.md`

**Назначение:** подробное резюме **только текущей сессии**.

Рекомендуемая структура:

```markdown
# Резюме сессии YYYY-MM-DD

## Контекст работы
...

## Выполненные задачи в этой сессии
...

## Важные изменения в коде
...

## Критические правила для следующего агента
...
```

**Важно:**

- Старые резюме из `SESSION_SUMMARY_*` регулярно переносить в `DONE_LIST_TRADETHERAPYBOT.md`.
- В текущем `SESSION_SUMMARY_*` должна быть информация **только об одной последней сессии**.

### 4. Дополнительные файлы

- `README.md` — общее описание бота.
- `README_FOR_NEXT_AGENT.md` — краткая инструкция для следующего агента.
- `CHANGELOG.md` — (опционально) сжатая история изменений по релизам.
- Папка `docs/` — база знаний:
  - `agent-onboarding.md` — онбординг.
  - `architecture.md` — архитектура и основные модули.
  - `business-rules.md` — бизнес‑правила (как работает продукт, роли пользователей и т.п.).
  - `patterns.md` — правила кода, паттерны и анти‑паттерны.
  - `database.md` — структура БД.
  - `security.md` — важные моменты по безопасности.
  - `specs/` — технические спеки по отдельным задачам.

## Правила начала новой сессии

Перед тем как что‑то менять в коде:

1. **Прочитать документацию проекта:**
   - `ROADMAP_TRADETHERAPYBOT.md` — понять планы и приоритеты.
   - `DONE_LIST_TRADETHERAPYBOT.md` — посмотреть, что уже сделано.
   - Текущий `SESSION_SUMMARY_YYYY-MM-DD.md` — статус последней сессии.
   - `README_FOR_NEXT_AGENT.md` — быстрый онбординг.
2. **Понять контекст задачи пользователя:**
   - что и зачем нужно поменять;
   - какие есть ограничения (дедлайны, риски, нельзя ломать прод и т.п.);
   - есть ли референсы/примеры поведения.
3. **Проверить связанные документы в `Main_docs/` (из `Cursor_Projects`), если работа идёт через агрегирующий репозиторий.**

## Workflow работы над задачей

### 1. Планирование

1. Создать или обновить спеку в `docs/specs/` по шаблону `spec-template.md`:
   - описание задачи и мотивация;
   - требования (функциональные и нефункциональные);
   - технические детали;
   - риски и как их снижать;
   - чек‑лист для самопроверки.
2. Задать пользователю вопросы, если что‑то неясно.
3. Оценить риски: что может сломаться, какие есть зависимости (БД, Telegram API, N8N и т.д.).

### 2. Реализация

1. **Проверить все связанные функции:**
   - перед изменением любой функции найти все места использования (поиском по коду);
   - учесть влияние на обработчики, callbacks, rate limiting и др.
2. **Следовать паттернам из `docs/patterns.md`:**
   - не изобретать велосипед;
   - переиспользовать существующие хелперы, `utils.py`, `validators.py` и т.п.
3. **Безопасность:**
   - **никогда** не использовать f‑строки в SQL‑запросах — только параметризованные запросы;
   - не хардкодить токены, пароли, chat_id и т.п.;
   - все чувствительные значения брать из `config.py` / env‑переменных.
4. **Логирование:**
   - использовать общий `logger` из `loader.py`;
   - логировать важные операции, ошибки, неожиданные ветки;
   - выбирать уровни логирования осознанно (INFO/WARNING/ERROR).
5. **Обработка ошибок:**
   - оборачивать внешние вызовы (Telegram API, БД, HTTP) в `try/except`;
   - при необходимости использовать retry‑механику (например, `retry_telegram_api`);
   - не «проглатывать» ошибки тихо — минимум логгировать.
6. **Тесты:**
   - писать unit‑тесты для новых функций и критичных участков;
   - обновлять/добавлять интеграционные тесты;
   - предлагать пользователю запустить `tests/` и `test_manual.py`, если он используется.

### 3. Проверка перед финалом

Перед тем как считать задачу законченной:

- [ ] Чек‑лист из спеки полностью выполнен.
- [ ] Проверены все связанные функции (поиск по коду).
- [ ] Все кнопки в меню имеют обработчики.
- [ ] Навигация между меню работает последовательно (нет «застреваний»).
- [ ] Добавлено/обновлено логирование.
- [ ] Нет хардкода секретов и временных заглушек в прод‑коде.
- [ ] Обновлены `SESSION_SUMMARY`, `DONE_LIST`, `ROADMAP` (где нужно).
- [ ] Линтер/форматтер не показывают ошибок.
- [ ] Удалены временные файлы и черновики.

**Критически важно:** не оставлять проект в состоянии, когда существующий функционал сломан.

## Правила работы с документацией

### При начале новой сессии

1. Прочитать:
   - `ROADMAP_TRADETHERAPYBOT.md`;
   - `DONE_LIST_TRADETHERAPYBOT.md`;
   - актуальный `SESSION_SUMMARY_YYYY-MM-DD.md`.
2. Понять, какие задачи уже решены, какие в процессе и какие запланированы.

### При завершении сессии

1. **Обновить `SESSION_SUMMARY_YYYY-MM-DD.md`:**
   - описать контекст, что делали в этой сессии;
   - перечислить выполненные задачи;
   - кратко зафиксировать изменения в коде (какие файлы/модули трогали);
   - записать важные замечания для следующего агента (подводные камни, временные решения).
2. **Обновить `DONE_LIST_TRADETHERAPYBOT.md`:**
   - перенести сюда информацию из предыдущего `SESSION_SUMMARY`;
   - добавить новые задачи с датами;
   - держать этот файл как «хронологию» развития проекта.
3. **Обновить `ROADMAP_TRADETHERAPYBOT.md`:**
   - отметить выполненные задачи (`- [x] Задача — ДАТА`);
   - скорректировать приоритеты оставшихся задач;
   - при необходимости добавить новые пункты.

### Важные принципы

- **Не плодить новые форматы документов** — использовать связку `ROADMAP + DONE_LIST + SESSION_SUMMARY + specs`.
- **Сохранять единообразие** структуры между всеми проектами (см. `Main_docs/RULES_CURSOR.md`).
- **Всегда читать существующие документы перед началом работы.**

## Правила коммитов

Подробные общие правила описаны в `Main_docs/COMMIT_CURSOR.md`. Для TradeTherapyBot действуют дополнительно следующие моменты:

### Что коммитится

✅ **Коммитятся в репозиторий `nikkronos/TradeTherapyBot`:**

- исходный код бота (Python‑файлы, миграции и т.п.);
- документация проекта (`README*`, `ROADMAP_TRADETHERAPYBOT.md`, `DONE_LIST_TRADETHERAPYBOT.md`, `SESSION_SUMMARY_*`, `docs/` и т.д.);
- настройки CI/CD и сервис‑файлы (например, systemd‑unit, requirements и др.), если они входят в проект.

❌ **Никогда не коммитятся:**

- `env_vars.txt`, `.env` и любые файлы с секретами;
- дампы баз данных, логи, временные файлы IDE;
- локальные скрипты/черновики, которые не нужны следующему разработчику.

### Процесс коммита

1. Убедиться, что:
   - все изменения протестированы;
   - проверены связанные функции;
   - нет ошибок линтера;
   - документация обновлена;
   - временные файлы удалены.
2. Подготовить осмысленное сообщение коммита:
   - `feat:` — новая функциональность;
   - `fix:` — исправление бага;
   - `docs:` — правки документации;
   - `refactor:` — рефакторинг без изменения поведения;
   - `test:` — добавление или правка тестов;
   - `chore:` — инфраструктура, конфиги и т.п.

Пример:

```text
feat: add rate limiting and receipt logging

- add per-user rate limiting helper
- add separate logger for ignored receipt messages
```

3. В `Cursor_Projects` агент формирует для пользователя готовые команды PowerShell для коммита/пуша; сами git‑команды выполняет пользователь.

## Завершение работы с задачей

Когда пользователь пишет, что все задачи выполнены (например: «Абсолютно все задачи выполнены.»), агент должен:

1. Ещё раз пробежаться по чек‑листу проверки.
2. Убедиться, что документация и код синхронизированы (нет противоречий).
3. Сформировать понятный финальный отчёт:
   - что было сделано;
   - какие изменения в коде;
   - какие риски остались/что можно сделать дальше;
   - какие команды нужно выполнить пользователю (если есть).

После этого, когда пользователь подтверждает («Закончили. Спасибо.»), задачу можно считать завершённой.
