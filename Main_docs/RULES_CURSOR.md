# Правила работы с Cursor и проектами

## Язык общения

- **Пользователь общается на русском языке** и базово общается на английском
- **AI агент может проводить всю внутреннюю работу на английском языке** (комментарии в коде, названия переменных, документация), так как это может сократить контекст
- **Общение с пользователем** - на русском языке

## Завершение работы

Обычно, когда пользователь видит, что контекст работы с агентом завершается, он понимает, что все задачи выполнены и озвучивает это: **"Абсолютно все задачи выполнены."**

**Как только пользователь пишет эту фразу, необходимо:**
1. Завершить все действия:
   - Обновить базу данных (если были изменения)
   - Обновить логи
   - Удалить ненужные временные файлы
   - Провести необходимый коммит (если проект коммитится)
   - Обновить документацию (SESSION_SUMMARY, DONE_LIST, ROADMAP)
2. Подготовить финальный отчёт о проделанной работе

**После завершения всех действий пользователь даст подтверждение финальной фразой: "Закончили. Спасибо."**

## GitHub репозитории

У пользователя есть **четыре репозитория** на GitHub:

1. **`nikkronos/Cursor_Projects`** — монорепозиторий на GitHub; локально папка **`Cursor_Projects`**: `Main_docs/`, `Projects/*` (кроме того, что в `.gitignore` и отдельных repo). В корне клона **`git remote origin`** → `…/Cursor_Projects.git`, не TradeTherapyBot (см. `Main_docs/COMMIT_CURSOR.md`, remote **`tradetherapybot`**). Старое имя на GitHub было `Cursor_Projectcs` — URL редиректит на `Cursor_Projects`.
2. **`nikkronos/TradeTherapyBot`** — код Telegram-бота TradeTherapyBot; **архивная копия** в монорепо: **`Projects/Non actual/TradeTherapyBot/`**. Пуш в репозиторий бота — из отдельного клона или из этой папки, если внутри настроен remote **`tradetherapybot`**.

## Терминал: PowerShell и выполнение команд

Пользователь работает в **Windows PowerShell**; пути часто содержат кириллицу и пробелы.

**Политика для агента (по умолчанию):**
1. **Сначала выполняй команды сам** через встроенный терминал Cursor (включая `git`, тесты, установку зависимостей, `ssh`/`scp`, если среда это позволяет).
2. **Обращайся к пользователю**, если команда системно не проходит (кодировка, права, интерактивный ввод, сеть, нестабильный shell, подозрение на опасное действие) или проще и надёжнее сделать шаг у пользователя на его машине — тогда дай **готовые команды**, ожидаемый результат и попроси прислать вывод при ошибке.

**Важно:**
- Указывай рабочую директорию явно; для путей с кириллицей используй кавычки, как в примере ниже.
- После неудачных попыток не зацикливайся — эскалируй с точными командами для копирования.

**Пример команд для пользователя (если агент передал шаг пользователю):**
```powershell
# Перейти в директорию проекта
cd "C:\Users\krono\OneDrive\Рабочий стол\Cursor_Projects\Projects\TradeTherapyBot"

# Выполнить команду
git status
```

Подробности по коммитам: `Main_docs/COMMIT_CURSOR.md`.

## Структура документов в каждом проекте

В каждой папке проекта должны быть следующие файлы:

### 1. ROADMAP_ИМЯПРОЕКТА.md
**Назначение:** Будущие цели и планы развития проекта

**Содержание:**
- Задачи в планах (с приоритетами)
- Отложенные задачи
- Идеи на будущее

**Пример:** `ROADMAP_PASTUHIBOT.md`, `ROADMAP_TRADETHERAPYBOT.md`

### 2. DONE_LIST_ИМЯПРОЕКТА.md
**Назначение:** Список выполненных задач (история)

**Содержание:**
- Все выполненные задачи из всех сессий
- Даты выполнения
- Краткое описание результатов

**Важно:** При новой сессии старые задачи из SESSION_SUMMARY переносятся сюда

**Пример:** `DONE_LIST_PASTUHIBOT.md`, `DONE_LIST_TRADETHERAPYBOT.md`

### 3. SESSION_SUMMARY_ДАТА.md
**Назначение:** Информация по последней сессии работы

**Содержание:**
- Контекст работы
- Выполненные задачи в этой сессии
- Изменения в коде
- Важные замечания для следующего агента
- Статус проекта

**Важно:** 
- Только текущая сессия
- При новой сессии старую информацию переносить в DONE_LIST
- Обновлять файл для новой сессии

**Пример:** `SESSION_SUMMARY_2025-12-23.md`

## Правила работы с документами

### При начале новой сессии:

1. **Прочитать:**
   - `Projects/ИМЯПРОЕКТА/ROADMAP_ИМЯПРОЕКТА.md` - что в планах
   - `Projects/ИМЯПРОЕКТА/DONE_LIST_ИМЯПРОЕКТА.md` - что уже сделано
   - `Projects/ИМЯПРОЕКТА/SESSION_SUMMARY_ДАТА.md` - последняя сессия

2. **Перед началом работы:**
   - Понять контекст проекта
   - Изучить выполненные задачи
   - Понять текущий статус

### При завершении сессии:

1. **Обновить SESSION_SUMMARY:**
   - Создать/обновить `Projects/ИМЯПРОЕКТА/SESSION_SUMMARY_ДАТА.md` с информацией о текущей сессии
   - Только текущая сессия, без старых данных

2. **Обновить DONE_LIST:**
   - Перенести информацию из предыдущего SESSION_SUMMARY в `Projects/ИМЯПРОЕКТА/DONE_LIST_ИМЯПРОЕКТА.md`
   - Добавить новые выполненные задачи
   - Обновить даты

3. **Обновить ROADMAP:**
   - Отметить выполненные задачи в `Projects/ИМЯПРОЕКТА/ROADMAP_ИМЯПРОЕКТА.md`
   - Добавить новые планы (если есть)
   - Обновить приоритеты

## Общие правила работы

### Перед началом работы:

1. **Изучить структуру проекта:**
   - Прочитать ROADMAP в папке `Projects/ИМЯПРОЕКТА/` - что в планах
   - Прочитать DONE_LIST в папке `Projects/ИМЯПРОЕКТА/` - что уже сделано
   - Прочитать последний SESSION_SUMMARY в папке `Projects/ИМЯПРОЕКТА/` - текущий статус

2. **Понять контекст:**
   - Что делает проект

### Во время работы:

1. **Следовать паттернам проекта:**
   - Использовать существующие паттерны кода
   - Следовать структуре проекта
   - Логировать важные операции

2. **Проверять связанные функции:**
   - Перед изменением функции найти все места использования
   - Проверить, что изменения не сломают существующую функциональность

### При завершении работы:

1. **Обновить документацию:**
   - SESSION_SUMMARY в `Projects/ИМЯПРОЕКТА/` - текущая сессия
   - DONE_LIST в `Projects/ИМЯПРОЕКТА/` - перенести старые задачи, добавить новые
   - ROADMAP в `Projects/ИМЯПРОЕКТА/` - обновить статус задач

2. **Проверить код:**
   - Все тесты проходят
   - Нет ошибок линтера
   - Код соответствует паттернам проекта

## Правила коммитов

**Подробная информация:** `Main_docs/COMMIT_CURSOR.md`

### Что коммитится:

✅ **Коммитятся (в `nikkronos/Cursor_Projects`):**
- Всё под **`Projects/Non actual/`** — архивные копии завершённых и неактуальных проектов
- `docs/`, `Main_docs/` (кроме `env_vars.txt`), корневые мета-файлы (`.gitignore`, `README.md`, `.github`, …)

❌ **Не в этом монорепо (см. `.gitignore` или отдельный репо):**
- `Projects/VPN/` — зеркалится в **`nikkronos/vpnservice`** (отдельный Git в этой папке)
- `TestN8N/` в корне — только локально
- `Main_docs/env_vars.txt` — секреты (НИКОГДА не коммитить!)
- `Projects/VPN/env_vars.txt` — секреты VPN (НИКОГДА не коммитить!)

**Отдельные репозитории (пуш не в `origin` монорепо):**
- `nikkronos/TradeTherapyBot` — из отдельного клона или из **`Projects/Non actual/TradeTherapyBot/`**, если там настроен remote
- `nikkronos/kopiya-iksuyemsya` — коммиты из **`Projects/Non actual/xxx/`**

### Процесс коммита:

1. **Коммиты (предпочтительно делает агент):**
   - Агент сам выполняет `git status`, `git add`, `git commit`, `git push` в нужной папке репозитория, если терминал и права это позволяют (см. `Main_docs/COMMIT_CURSOR.md`).
   - Если автоматический коммит/пуш невозможен или нестабилен — агент даёт пользователю **готовый блок команд** и кратко объясняет, что должно получиться.

2. **Перед коммитом:**
   - ✅ Все изменения протестированы
   - ✅ Проверены связанные функции
   - ✅ Нет ошибок линтера
   - ✅ Документация обновлена
   - ✅ Временные файлы удалены

### Типы коммитов:

- `feat:` - новая функциональность
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `refactor:` - рефакторинг кода
- `test:` - добавление/изменение тестов
- `chore:` - обновление зависимостей, настройка

**Пример:** `docs: унифицирована структура документов и созданы правила работы`

## Важные напоминания

1. **Не создавать лишних документов** - использовать только ROADMAP, DONE_LIST, SESSION_SUMMARY
2. **Актуализировать документы** - старую информацию переносить в DONE_LIST
3. **Следовать структуре** - одинаковые файлы во всех проектах
4. **Читать перед началом** - всегда начинать с изучения существующих документов
5. **Проверять .gitignore** — что **`Projects/VPN/`** и **`TestN8N/`** по-прежнему исключены из монорепо; архив **`Projects/Non actual/`** наоборот **трекается**
