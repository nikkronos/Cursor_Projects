# Правила работы с Cursor и проектами

## Язык общения

- **Пользователь общается на русском языке** и базово общается на английском
- **AI агент может проводить всю внутреннюю работу на английском языке** (комментарии в коде, названия переменных, документация), так как это может сократить контекст
- **Общение с пользователем** - на русском языке

## Завершение работы

Обычно, когда пользователь видит, что контекст работы с агентом завершается, он понимает, что все задачи выполнены и озвучивает это: **"Абсолютно все задачи выполнены."**

**Как только пользователь пишет эту фразу, необходимо:**
1. Завершить все действия:
   - Обновить базу данных (если были изменения)
   - Обновить логи
   - Удалить ненужные временные файлы
   - Провести необходимый коммит (если проект коммитится)
   - Обновить документацию (SESSION_SUMMARY, DONE_LIST, ROADMAP)
2. Подготовить финальный отчёт о проделанной работе

**После завершения всех действий пользователь даст подтверждение финальной фразой: "Закончили. Спасибо."**

## Сервер Timeweb

- У пользователя есть оплаченный сервер на Timeweb
- На сервере работает TradeTherapyBot (создан с AI агентами)
- На сервере работает PastuhiBot (создан с разработчиком, копия с сервера)
- **Важно:** Все секреты (пароли, токены, API ключи) хранятся в файле `env_vars.txt` (локально) или `.env` (на сервере)
- Файлы с секретами **НИКОГДА** не коммитятся в Git
- Используется `.gitignore` для исключения секретных файлов
- Подробная документация о сервере: `docs/server-timeweb.md`

## GitHub репозиторий

- У пользователя есть подключённый репозиторий на GitHub: `nikkronos/TradeTherapyBot`
- Вся логика должна быть отображена в документах, чтобы все AI агенты всё понимали
- Если нужно переименовать что-то на GitHub, можно сделать, чтобы всё адекватно и логично синхронизировалось
- Настроен автоматический CI/CD деплой через GitHub Actions
- Коммиты и деплои обычно агент делает автоматически, но иногда обращается к пользователю

## Работа через PowerShell

Пользователь выполняет задачи через PowerShell со своего компьютера (Windows).

**Важно для агента:**
- При необходимости выполнения команд в терминале, агент должен давать **пошаговые команды с нужными директориями**
- Учитывать, что пользователь работает в Windows PowerShell
- Пути должны быть указаны явно
- Если есть проблемы с кодировкой или путями (особенно с русскими символами), агент должен предоставить альтернативные решения

**Пример правильных команд:**
```powershell
# Перейти в директорию проекта
cd "C:\Users\krono\OneDrive\Рабочий стол\Cursor_test\TradeTherapyBot"

# Выполнить команду
git status
```

## Структура документов в каждом проекте

В каждой папке проекта должны быть следующие файлы:

### 1. ROADMAP_ИМЯПРОЕКТА.md
**Назначение:** Будущие цели и планы развития проекта

**Содержание:**
- Задачи в планах (с приоритетами)
- Отложенные задачи
- Идеи на будущее

**Пример:** `ROADMAP_PASTUHIBOT.md`, `ROADMAP_TRADETHERAPYBOT.md`

### 2. DONE_LIST_ИМЯПРОЕКТА.md
**Назначение:** Список выполненных задач (история)

**Содержание:**
- Все выполненные задачи из всех сессий
- Даты выполнения
- Краткое описание результатов

**Важно:** При новой сессии старые задачи из SESSION_SUMMARY переносятся сюда

**Пример:** `DONE_LIST_PASTUHIBOT.md`, `DONE_LIST_TRADETHERAPYBOT.md`

### 3. SESSION_SUMMARY_ДАТА.md
**Назначение:** Информация по последней сессии работы

**Содержание:**
- Контекст работы
- Выполненные задачи в этой сессии
- Изменения в коде
- Важные замечания для следующего агента
- Статус проекта

**Важно:** 
- Только текущая сессия
- При новой сессии старую информацию переносить в DONE_LIST
- Обновлять файл для новой сессии

**Пример:** `SESSION_SUMMARY_2025-12-23.md`

## Правила работы с документами

### При начале новой сессии:

1. **Прочитать:**
   - `ROADMAP_ИМЯПРОЕКТА.md` - что в планах
   - `DONE_LIST_ИМЯПРОЕКТА.md` - что уже сделано
   - `SESSION_SUMMARY_ДАТА.md` - последняя сессия

2. **Перед началом работы:**
   - Понять контекст проекта
   - Изучить выполненные задачи
   - Понять текущий статус

### При завершении сессии:

1. **Обновить SESSION_SUMMARY:**
   - Создать/обновить `SESSION_SUMMARY_ДАТА.md` с информацией о текущей сессии
   - Только текущая сессия, без старых данных

2. **Обновить DONE_LIST:**
   - Перенести информацию из предыдущего SESSION_SUMMARY в DONE_LIST
   - Добавить новые выполненные задачи
   - Обновить даты

3. **Обновить ROADMAP:**
   - Отметить выполненные задачи
   - Добавить новые планы (если есть)
   - Обновить приоритеты

## Общие правила работы

### Перед началом работы:

1. **Изучить структуру проекта:**
   - Прочитать ROADMAP - что в планах
   - Прочитать DONE_LIST - что уже сделано
   - Прочитать последний SESSION_SUMMARY - текущий статус

2. **Понять контекст:**
   - Что делает проект
   - Какие технологии используются
   - Какие есть ограничения

### Во время работы:

1. **Следовать паттернам проекта:**
   - Использовать существующие паттерны кода
   - Следовать структуре проекта
   - Логировать важные операции

2. **Проверять связанные функции:**
   - Перед изменением функции найти все места использования
   - Проверить, что изменения не сломают существующую функциональность

### При завершении работы:

1. **Обновить документацию:**
   - SESSION_SUMMARY - текущая сессия
   - DONE_LIST - перенести старые задачи, добавить новые
   - ROADMAP - обновить статус задач

2. **Проверить код:**
   - Все тесты проходят
   - Нет ошибок линтера
   - Код соответствует паттернам проекта

## Правила коммитов

**Подробная информация:** `COMMIT_CURSOR.md`

### Что коммитится:

✅ **Коммитятся:**
- Все файлы из корня `CURSOR_TEST`
- Проект `TradeTherapyBot/` (полностью)
- Документация в корне (`docs/`, `RULES_CURSOR.md`, `QUICK_START_AGENT.md` и т.д.)
- Все общие файлы и настройки

❌ **НЕ коммитятся:**
- `ParentChildResearch/` - временный исследовательский проект
- `PastuhiBot/` - локальная копия боевого бота (хранится только локально)

### Процесс коммита:

1. **Автоматический коммит (предпочтительно):**
   - Агент может делать коммиты автоматически после тестирования
   - Выполняет: `git add .`, `git commit -m "..."`, `git push origin main`

2. **Ручной коммит:**
   - Если автоматический коммит невозможен, агент предоставляет команды для ручного выполнения

3. **Перед коммитом:**
   - ✅ Все изменения протестированы
   - ✅ Проверены связанные функции
   - ✅ Нет ошибок линтера
   - ✅ Документация обновлена
   - ✅ Временные файлы удалены

### Типы коммитов:

- `feat:` - новая функциональность
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `refactor:` - рефакторинг кода
- `test:` - добавление/изменение тестов
- `chore:` - обновление зависимостей, настройка

**Пример:** `docs: унифицирована структура документов и созданы правила работы`

## Важные напоминания

1. **Не создавать лишних документов** - использовать только ROADMAP, DONE_LIST, SESSION_SUMMARY
2. **Актуализировать документы** - старую информацию переносить в DONE_LIST
3. **Следовать структуре** - одинаковые файлы во всех проектах
4. **Читать перед началом** - всегда начинать с изучения существующих документов
5. **Проверять .gitignore** - убедиться, что ParentChildResearch и PastuhiBot исключены

## Исключения

- **База знаний** (`docs/`) - может быть в каждом проекте, это нормально
- **README.md** - общее описание проекта, может быть
- **CHANGELOG.md** - история изменений, может быть
- **Спецификации** (`docs/specs/`) - технические задания, могут быть

Но основные файлы для работы агента: **ROADMAP**, **DONE_LIST**, **SESSION_SUMMARY**
