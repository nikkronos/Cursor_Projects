# Правила безопасности

## Секреты и конфиденциальные данные

### Основные правила
- **НИКОГДА** не хардкодить пароли, токены, API-ключи в коде
- **НИКОГДА** не писать секреты в чате с AI агентом (даже в примерах кода)
- **НИКОГДА** не добавлять секреты в логи
- **НИКОГДА** не коммитить секреты в Git

### Хранение секретов

#### Локальная разработка
- Все секреты хранить в файле `env_vars.txt`
- Файл `env_vars.txt` должен быть в `.gitignore`
- Использовать `env_vars.example.txt` для примера структуры (без реальных значений)
- Загружать секреты через `config.py` с помощью `python-dotenv`

**Структура `env_vars.txt`:**
```
BOT_TOKEN=ваш_токен_бота
ADMIN_ID=ваш_telegram_id
GROUP_CHAT_ID=id_закрытой_группы
```

#### Продакшн-окружение (GitHub Actions)
Для CI/CD и автоматических деплоев секреты хранятся в GitHub Secrets:
1. Перейти в репозиторий на GitHub
2. Settings → Secrets and variables → Actions
3. Нажать "New repository secret"
4. Добавить секрет (например, `BOT_TOKEN`)
5. В GitHub Actions использовать через `${{ secrets.BOT_TOKEN }}`

**Пример использования в GitHub Actions:**
```yaml
env:
  BOT_TOKEN: ${{ secrets.BOT_TOKEN }}
  ADMIN_ID: ${{ secrets.ADMIN_ID }}
  GROUP_CHAT_ID: ${{ secrets.GROUP_CHAT_ID }}
```

### Проверка безопасности
Перед каждым коммитом проверять:
- [ ] Нет секретов в коде
- [ ] Нет секретов в комментариях
- [ ] Нет секретов в истории Git (если случайно закоммитили - использовать `git filter-branch` или BFG Repo-Cleaner)

## SQL Injection защита

### Обязательные правила
- **ВСЕГДА** использовать параметризованные запросы
- **НИКОГДА** не форматировать SQL-запросы через f-strings или `.format()`

**Правильно:**
```python
cursor.execute("SELECT * FROM users WHERE telegram_id = ?", (user_id,))
```

**Неправильно:**
```python
cursor.execute(f"SELECT * FROM users WHERE telegram_id = {user_id}")  # ОПАСНО!
```

## Валидация входных данных

### Правила валидации
- Все данные от пользователей должны проверяться перед использованием
- Проверять типы данных (int, str, bool)
- Проверять диапазоны значений (например, user_id должен быть положительным числом)
- Проверять наличие обязательных полей
- Ограничивать длину строковых значений

**Пример валидации:**
```python
def validate_user_id(user_id):
    """Валидация ID пользователя"""
    try:
        user_id = int(user_id)
        if user_id <= 0:
            raise ValueError("User ID must be positive")
        return user_id
    except (ValueError, TypeError):
        raise ValueError(f"Invalid user ID: {user_id}")
```

## Защита от злоупотреблений

### Rate Limiting
Для защиты от спама и злоупотреблений (в будущем):
- Ограничивать количество запросов от одного пользователя
- Ограничивать количество сообщений в единицу времени
- Блокировать подозрительную активность

### Проверка прав доступа
- Административные команды доступны только администратору
- Проверять `message.from_user.id == ADMIN_ID` перед выполнением админ-команд
- Логировать попытки несанкционированного доступа

## Безопасность базы данных

### Резервное копирование
- Регулярно делать бэкапы БД (команда `/backup`)
- Хранить бэкапы в безопасном месте
- Шифровать бэкапы при хранении в облаке

### Права доступа
- Файл `bot.db` должен иметь ограниченные права доступа на сервере
- База данных должна быть доступна только процессу бота

## Логирование без секретов

### Правила логирования
- В логах не должно быть токенов, паролей, API-ключей
- При необходимости логировать только частичное значение (например, `BOT_TOKEN=***123`)

**Правильно:**
```python
logger.info(f"Bot token loaded: {'*' * 20}")
logger.info(f"User {user_id} authenticated")
```

**Неправильно:**
```python
logger.info(f"Bot token: {BOT_TOKEN}")  # ОПАСНО!
```

## Обновление зависимостей

### Безопасность пакетов
- Регулярно обновлять зависимости (`pip install --upgrade`)
- Проверять уязвимости в зависимостях (например, через `pip-audit` или GitHub Dependabot)
- Использовать проверенные версии библиотек

## Чеклист безопасности перед деплоем
- [ ] Нет секретов в коде
- [ ] Все SQL-запросы параметризованы
- [ ] Входные данные валидируются
- [ ] Права доступа проверяются
- [ ] Логи не содержат секретов
- [ ] Зависимости обновлены
- [ ] База данных защищена (права доступа)
- [ ] Бэкапы настроены
