## 🔄 Workflow работы

### 1. Планирование

После понимания задачи:

1. **Создай спецификацию** в `docs/specs/` используя `spec-template.md`
   - Описание задачи
   - Требования
   - Технические детали
   - Риски и митигация
   - Чеклист для проверки

2. **Задай вопросы** пользователю:
   - Уточни неясные моменты
   - Предложи альтернативные решения
   - Обсуди риски

3. **Проведи самооценку:**
   - Какие есть риски?
   - Какие альтернативные решения?
   - Какие зависимости?
   - Что может сломаться?

### 2. Реализация

1. **Проверь связанные функции:**
   - Используй `grep` или `codebase_search` для поиска всех мест использования
   - Убедись, что изменения не сломают существующий код

2. **Следуй паттернам проекта:**
   - Используй существующие паттерны из `docs/patterns.md`
   - Следуй структуре проекта
   - Используй валидацию из `validators.py` (если есть)

3. **Добавь логирование:**
   - Логируй все важные операции
   - Используй правильные уровни логирования (INFO, WARNING, ERROR)

4. **Обрабатывай ошибки:**
   - Всегда используй `try-except`
   - Логируй ошибки
   - Предоставляй понятные сообщения пользователю

5. **Пиши тесты:**
   - Unit-тесты для новых функций
   - Интеграционные тесты для критических путей
   - Предложи пользователю запустить `test_manual.py`

### 3. Проверка

Перед завершением проверь:

- [ ] Чеклист из спека выполнен?
- [ ] Проверены все связанные функции?
- [ ] Все кнопки имеют обработчики?
- [ ] Навигация между меню работает?
- [ ] Все тесты написаны?
- [ ] Соответствует `patterns.md`?
- [ ] Логирование добавлено?
- [ ] Нет hardcoded секретов?
- [ ] Документация обновлена?
- [ ] Линтер не находит ошибок?
- [ ] Временные файлы удалены?

**КРИТИЧЕСКИ ВАЖНО:** Перед коммитом убедись, что не сломались существующие функции!

## 💻 Правила работы с кодом

### Безопасность

1. **Никогда не используй f-строки в SQL-запросах!**
   - Используй параметризованные запросы
   - Валидируй все входные данные

2. **Никогда не хардкоди секреты!**
   - Используй переменные окружения
   - Читай из `config.py` или `env_vars.txt`

3. **Валидируй все входные данные:**
   - Используй `validators.py` (если есть)
   - Проверяй типы данных
   - Проверяй диапазоны значений

### Качество кода

1. **Следуй паттернам проекта:**
   - Изучи `docs/patterns.md`
   - Используй существующие паттерны
   - Не изобретай велосипед

2. **Проверяй связанные функции:**
   - Перед изменением функции найди все места использования
   - Убедись, что изменения не сломают существующий код
   - Используй `grep` или `codebase_search`

3. **Обновляй UI при изменении логики:**
   - Если изменилась логика, обнови соответствующие кнопки
   - Проверь навигацию между меню
   - Убедись, что все кнопки имеют обработчики

4. **Логируй важные операции:**
   - Используй `logger` из `loader.py`
   - Правильные уровни: INFO, WARNING, ERROR, CRITICAL
   - Логируй входные параметры и результаты

### Обработка ошибок

1. **Всегда используй try-except:**
   - Обрабатывай все возможные ошибки
   - Логируй ошибки
   - Предоставляй понятные сообщения

2. **Используй retry для внешних API:**
   - Telegram API может быть нестабильным
   - Используй `retry_telegram_api` из `utils.py` (если есть)

3. **Rate limiting:**
   - Соблюдай лимиты Telegram API
   - Используй `rate_limit` из `utils.py` (если есть)

## 📝 Правила работы с документацией

### При начале новой сессии:

1. **Прочитай:**
   - `ROADMAP_ИМЯПРОЕКТА.md` - что в планах
   - `DONE_LIST_ИМЯПРОЕКТА.md` - что уже сделано
   - `SESSION_SUMMARY_ДАТА.md` - последняя сессия

2. **Пойми контекст:**
   - Что делает проект
   - Какие технологии используются
   - Какие есть ограничения

### При завершении сессии:

1. **Обнови SESSION_SUMMARY:**
   - Создай/обнови `SESSION_SUMMARY_ДАТА.md`
   - Только текущая сессия, без старых данных
   - Включи: контекст, выполненные задачи, изменения в коде, важные замечания

2. **Обнови DONE_LIST:**
   - Перенеси информацию из предыдущего SESSION_SUMMARY в DONE_LIST
   - Добавь новые выполненные задачи
   - Обнови даты

3. **Обнови ROADMAP:**
   - Отметь выполненные задачи
   - Добавь новые планы (если есть)
   - Обнови приоритеты

### Важные правила:

- **Не создавай лишних документов** - используй только ROADMAP, DONE_LIST, SESSION_SUMMARY
- **Актуализируй документы** - старую информацию переноси в DONE_LIST
- **Следуй структуре** - одинаковые файлы во всех проектах
- **Читай перед началом** - всегда начинай с изучения существующих документов

1. **Обнови SESSION_SUMMARY:**
   ```markdown

   [Описание контекста]
   ```

2. **Обнови DONE_LIST:**
   - Перенеси старую информацию из SESSION_SUMMARY

   - Обнови статус проекта

   - Отметь выполненные задачи: `- [x] Задача - ДАТА`
   - Обнови приоритеты

### Git (если проект коммитится)

Для проектов, которые коммитятся в Git (например, TradeTherapyBot):

```bash
# Перейти в папку проекта
cd TradeTherapyBot

# Добавить изменения
git add .

# Сделать коммит
git commit -m "feat: описание изменений

- Детали изменений
- Что было исправлено
- Что было добавлено"

# Отправить в репозиторий
git push origin main
```

**Важно:** Для PastuhiBot изменения НЕ коммитятся - хранятся локально!

## 🗂️ Проекты в репозитории

### TradeTherapyBot

**Описание:** Telegram-бот для автоматизации управления доступом к закрытому сообществу трейдеров "Trade Therapy".

**Особенности:**
- Коммитится в Git
- Автоматический CI/CD деплой
- Работает на сервере 24/7

**Документы:**
- `ROADMAP_TRADETHERAPYBOT.md`
- `DONE_LIST_TRADETHERAPYBOT.md`
- `SESSION_SUMMARY_ДАТА.md`

**База знаний:** `TradeTherapyBot/docs/`

### PastuhiBot

**Описание:** Telegram-бот для копирования содержимого определенных Telegram каналов.

**Особенности:**
- **НЕ коммитится в Git** - все изменения локально
- Боевой бот работает на сервере Timeweb
- Изменения применяются вручную после тестирования

**Документы:**
- `ROADMAP_PASTUHIBOT.md`
- `DONE_LIST_PASTUHIBOT.md`
- `SESSION_SUMMARY_ДАТА.md`

**База знаний:** `PastuhiBot/docs/`

### Копия иксуюемся

**Описание:** Telegram-бот для копирования информации из одного Telegram-канала в другой (бот-перелив).

**Особенности:**
- **Коммитится в Git** - все изменения через Git (в отдельную папку)
- Создан по аналогии с PastuhiBot
- Работает на сервере Timeweb (тот же сервер, что и TradeTherapyBot и PastuhiBot)
- Создан с помощью AI агентов

**Документы:**
- `ROADMAP_КОПИЯ_ИКСУЮЕМСЯ.md`
- `DONE_LIST_КОПИЯ_ИКСУЮЕМСЯ.md`
- `SESSION_SUMMARY_ДАТА.md`

**База знаний:** `xxx/docs/` (будет создана)

**Технологии:**
- aiogram 2.25.1 или pyTelegramBotAPI (для бота управления)
- Pyrogram 2.0.106 (для юзербота)
- SQLite
- APScheduler (если потребуется)
