# Техническое задание: Настройка автоматического CI/CD для деплоя на сервер

## Что надо сделать
Настроить автоматическое обновление бота на сервере при каждом push в основную ветку GitHub репозитория. После успешного прохождения тестов, код должен автоматически деплоиться на сервер и перезапускаться через systemd service.

## Как это должно работать
- При каждом `push` в ветку `main` (или `master`) автоматически запускается workflow
- Сначала выполняются тесты (уже настроено)
- После успешного прохождения тестов выполняется деплой на сервер:
  1. Подключение к серверу по SSH
  2. Обновление кода из GitHub (git pull)
  3. Установка/обновление зависимостей (если изменился requirements.txt)
  4. Перезапуск systemd service (`tradetherapybot.service`)
  5. Проверка статуса сервиса
- Если деплой успешен - уведомление администратора (опционально)
- Если деплой неудачен - уведомление об ошибке и откат к предыдущей версии

## Какие инструменты нам понадобятся
- GitHub Actions (встроенный CI/CD сервис GitHub)
- SSH доступ к серверу (для деплоя)
- GitHub Secrets (для хранения SSH ключей и данных сервера)
- Git на сервере (для git pull)
- systemd (уже используется для управления ботом)
- Стандартные GitHub Actions (actions/checkout, actions/setup-python)
- SSH action для GitHub Actions (например, appleboy/ssh-action)

## Объяснение для пользователя

### Что такое деплой (deployment)?
**Деплой** - это процесс обновления кода бота на сервере. Сейчас вы делаете это вручную:
1. Меняете код на компьютере
2. Загружаете файлы на сервер через `scp` (или вручную)
3. Подключаетесь к серверу по SSH
4. Перезапускаете бота командой `systemctl restart tradetherapybot.service`

**Автоматический деплой** делает все это автоматически:
1. Вы делаете `git push` в GitHub (отправляете изменения в облако)
2. GitHub автоматически:
   - Запускает тесты
   - Если тесты прошли → обновляет код на сервере
   - Перезапускает бота

**Преимущества:** Не нужно вручную загружать файлы и перезапускать бота. Все происходит автоматически после того, как вы отправили код в GitHub.

### Различия веток Git (main vs другие)
В Git есть понятие **веток** (branches) - это разные версии вашего кода.

**`main` (или `master`)** - это основная ветка, где находится рабочий код, который работает на сервере.

**Другие ветки** (например, `develop`, `feature/new-function`) - это экспериментальные ветки для разработки новых функций.

**Что выбрать для автоматического деплоя?**
- ✅ **`main`** - деплой из этой ветки означает, что каждое изменение, которое вы отправите в `main`, автоматически попадет на сервер
- ⚠️ **Другие ветки** - можно настроить деплой из других веток для тестирования

## Потенциальные проблемы и риски
- **Риск 1:** SSH ключ может быть скомпрометирован
  - **Митигация:** Использовать отдельный SSH ключ только для деплоя, с ограниченными правами. Регулярно менять ключи
- **Риск 2:** Деплой может сломать работающего бота
  - **Митигация:** Всегда делать бэкап БД перед деплоем. Настроить откат при ошибках. Тестировать изменения локально перед деплоем
- **Риск 3:** Проблемы с зависимостями при обновлении requirements.txt
  - **Митигация:** Проверять изменения requirements.txt, тестировать установку зависимостей локально
- **Риск 4:** Конфликты при git pull (локальные изменения на сервере)
  - **Митигация:** Использовать `git reset --hard` для чистого обновления. Или проверить статус git перед pull
- **Риск 5:** Бот может не запуститься после обновления из-за ошибок в коде
  - **Митигация:** Тесты должны проходить перед деплоем. Настроить откат. Проверять статус service после перезапуска
- **Риск 6:** Недостаточно прав для перезапуска systemd service
  - **Митигация:** Настроить sudo для команды systemctl или использовать пользователя с правами на управление service
- **Риск 7:** Долгий деплой может прервать работу бота
  - **Митигация:** Оптимизировать процесс деплоя. Использовать graceful restart если возможно
