# Работа с Git и GitHub

## 1. Первоначальная настройка (выполняется один раз)

### Шаг 1: Инициализация Git в папке проекта
Откройте терминал в папке проекта (`C:\Users\krono\OneDrive\Рабочий стол\Cursor_test`) и выполните:
```bash
git init
```

### Шаг 2: Настройка имени пользователя и почты (если не делали раньше)
```bash
git config --global user.name "Ваше Имя"
git config --global user.email "ваша@почта.com"
```

### Шаг 3: Создание репозитория на GitHub
1. Зайдите на [GitHub](https://github.com) и войдите в аккаунт.
2. Нажмите кнопку **"New"** (создать репозиторий).
3. Назовите его `TradeTherapyBot`.
4. Оставьте его **Private** (так как там может быть код приватного бота).
5. **Не ставьте** галочки "Add a README file", ".gitignore" (у нас они уже есть).
6. Нажмите **Create repository**.

### Шаг 4: Связывание локального проекта с GitHub
Скопируйте ссылку на репозиторий (вида `https://github.com/ваш_ник/TradeTherapyBot.git`) и выполните в терминале:
```bash
git remote add origin https://github.com/ваш_ник/TradeTherapyBot.git
git branch -M main
```

## 2. Сохранение изменений (Ежедневный цикл)

Каждый раз, когда вы завершили задачу или этап работы:

### 1. Проверить статус (что изменилось)
```bash
git status
```

### 2. Добавить файлы в индекс (подготовка к сохранению)
```bash
git add .
```
*(Точка означает "добавить все файлы").*

### 3. Сделать коммит (сохранение с описанием)
```bash
git commit -m "Описание того, что сделали (например: добавил защиту от входа без подписки)"
```

### 4. Отправить в облако (на GitHub)
```bash
git push -u origin main
```

**После push:** Если настроен автоматический CI/CD деплой, код автоматически обновится на сервере после успешных тестов. См. `QUICK_START_CICD.md`

**💡 Важно:** Регулярные коммиты и push предотвращают потерю файлов. Подробнее см. `docs/preventing-file-loss.md`

## 3. Если что-то сломалось (Откат)

Если вы сделали что-то не то и хотите вернуть файлы к состоянию последнего коммита:
```bash
git checkout .
```
*(Осторожно! Это удалит все несохраненные изменения).*

## 4. Предотвращение потери файлов

### ⚠️ Важно: Регулярные коммиты

**Главное правило:** Коммитьте часто! После каждой законченной задачи:
```bash
git add .
git commit -m "описание изменений"
git push origin main
```

**Почему это важно:**
- Git сохраняет полную историю всех изменений
- Можно восстановить любую версию файла
- GitHub служит резервной копией в облаке
- Если файл случайно очистился - можно восстановить из истории

### ✅ Чеклист перед коммитом

Перед каждым коммитом проверяйте:
- [ ] `git status` - посмотреть, что изменено
- [ ] Все важные файлы не пустые (проверить ключевые файлы)
- [ ] Код работает (запустить бота или тесты)
- [ ] Нет случайных изменений
- [ ] Понятное описание коммита

### 🔍 Проверка пустых файлов

Перед коммитом проверьте, нет ли случайно очищенных файлов:
```powershell
# Проверить важные Python файлы
Get-ChildItem -Recurse -File *.py | ForEach-Object {
    $lines = (Get-Content $_.FullName).Count
    if ($lines -lt 5) {
        Write-Host "⚠️  ВНИМАНИЕ: $($_.FullName) содержит только $lines строк!"
    }
}
```

### 🔧 Восстановление файла из истории

Если файл случайно очистился или удалился:

1. **Найти коммит, где файл был:**
   ```bash
   git log --all --full-history -- <путь_к_файлу>
   ```

2. **Восстановить файл:**
   ```bash
   git checkout <commit-hash> -- <путь_к_файлу>
   ```

3. **Закоммитить восстановление:**
   ```bash
   git add <путь_к_файлу>
   git commit -m "fix: восстановление <имя_файла> из истории Git"
   git push origin main
   ```

### 💡 Рекомендации

1. **Коммитьте после каждого значимого изменения** - лучше много маленьких коммитов
2. **Push сразу после коммита** - GitHub будет резервной копией
3. **Используйте ветки для экспериментов** - работайте в отдельной ветке, затем сливайте в main
4. **Проверяйте изменения перед коммитом** - `git diff` покажет все изменения
5. **Будьте осторожны с массовыми операциями** - перед "Replace All" или массовым удалением делайте коммит

### 🚨 Что делать, если файл потерян

1. Проверить историю Git: `git log --all --full-history -- <файл>`
2. Восстановить из истории: `git checkout <hash> -- <файл>`
3. Если файла нет в Git - взять с сервера (если бот там работает)
4. Проверить GitHub - возможно файл в последней версии на сервере
