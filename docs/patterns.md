# Паттерны и Стандарты кода

## Общие правила
- Все настройки (токены, пароли) выносятся в `env_vars.txt` и загружаются через `config.py`.
- Используется **SQLite** через стандартную библиотеку `sqlite3`. Важно использовать параметризованные запросы (`?`), чтобы избежать SQL-инъекций.
- Используется **Context Manager** (`with get_db_connection() as conn:`) для работы с БД, чтобы гарантировать закрытие соединений.
- Логирование через модуль `logging`. `print()` не используется для продакшн-логов.

## Обработка ошибок
- Все вызовы API Telegram и БД оборачиваются в блоки `try-except`.
- Ошибки логируются с уровнем `ERROR` или `CRITICAL`.
- При ошибке отправки сообщения пользователю, работа бота не должна прерываться.
- Для критичных операций Telegram API используется `retry_telegram_api()` из `utils.py` для автоматических повторов при временных ошибках.
- Для отправки сообщений используется `safe_send_message()` из `utils.py`, которая автоматически обрабатывает ошибки и retry.

### Работа с Telegram API для получения данных пользователей
- **`get_chat(user_id)`** - работает только если пользователь писал боту в личку или бот может видеть пользователя
- **`get_chat_member(chat_id, user_id)`** - работает если:
  - Бот является администратором группы (даже без особых прав)
  - Пользователь находится в группе
  - Бот имеет доступ к информации о членах группы
- **Ошибка 401 Unauthorized** при `get_chat_member()` обычно означает:
  - Бот не является администратором группы (проверить в настройках группы)
  - Неправильный `GROUP_CHAT_ID` (проверить формат: для групп начинается с `-100`)
  - Токен неверный или не имеет прав (проверить токен на сервере)
- **Важно:** При запуске скриптов локально убедиться, что используется тот же токен, что и на сервере

## Структура проекта
Проект разделен на модули (`handlers.py`, `services.py`, `database.py`) для разделения ответственности.

## Структура хендлеров
- Используются декораторы `@bot.message_handler`.
- Хендлеры разделены по смыслу (админские, пользовательские, колбэки) внутри `handlers.py` (в будущем можно разделить на файлы).
- Для многошаговых диалогов используется `bot.register_next_step_handler`.

## Безопасность

### Секреты и конфиденциальные данные
- **НИКОГДА** не хардкодить пароли, токены, API-ключи в коде
- **НИКОГДА** не писать секреты в чате с агентом (даже в примерах)
- **НИКОГДА** не добавлять секреты в логи
- Все секреты хранить в `env_vars.txt` (который в `.gitignore`)
- Файл `.env` и `env_vars.txt` всегда должны быть в `.gitignore`
- Использовать только `env_vars.example.txt` для примера структуры

Подробные правила безопасности см. в `docs/security.md`.

### Валидация входных данных
- Все данные от пользователей должны валидироваться перед записью в БД
- Использовать параметризованные SQL-запросы (всегда `?`, никогда не форматировать строки)
- Проверять типы данных и диапазоны значений

## Логирование

### Правила логирования
- **Логировать все важные операции:**
  - Действия администратора (изменение подписок, удаление пользователей)
  - Ошибки подключения к БД
  - Ошибки API Telegram
  - Проверки подписок и автоматические действия
  - Подтверждение/отклонение оплат
  - Вход/выход пользователей из группы
  - Изменения статусов подписок

### Уровни логирования
- `DEBUG` - детальная информация для отладки
- `INFO` - общая информация о работе бота
- `WARNING` - предупреждения (некритичные проблемы)
- `ERROR` - ошибки, которые не прерывают работу
- `CRITICAL` - критические ошибки (БД недоступна, бот не может запуститься)

### Формат логов
```python
logger.info(f"Подписка пользователя {user_id} продлена до {end_date}")
logger.error(f"Ошибка при удалении пользователя {user_id}: {e}")
logger.warning(f"Пользователь {user_id} не найден в группе")
```

**Важно:** В логах не должно быть секретов (токены, пароли, API-ключи).

## Тестирование

### Обязательные правила
- **На каждую функцию пишутся тесты**
- Тесты планируются еще на этапе ТЗ (в spec-template.md)
- Тесты запускаются после каждой задачи
- Тесты должны проходить перед коммитом

### Структура тестов
- Тесты хранятся в папке `tests/`
- Имя файла: `test_<имя_модуля>.py`
- Использовать библиотеку `pytest`

### Пример теста
```python
def test_add_subscription_days():
    """Тест добавления дней к подписке"""
    # Arrange
    user_id = 123456
    days_to_add = 30
    
    # Act
    result = add_subscription_days_logic(user_id, days_to_add, admin_chat_id)
    
    # Assert
    user = get_user_status(user_id)
    assert user['subscription_status'] == 'active'
    assert user['subscription_end_date'] is not None
```

### Автоматический запуск тестов
- Настроен GitHub Actions для автоматического запуска тестов при каждом коммите (в планах)
- Если тесты не проходят - GitHub оповестит об этом

## Self-Review (Самопроверка агента)

### Перед началом работы
1. Оценить план решения:
   - Это лучшее решение? Почему?
   - Какие подводные камни?
   - Какие риски?
   - Альтернативные решения?

2. Для каждой задачи определить:
   - Критерии успеха
   - Чеклист самопроверки

3. **КРИТИЧЕСКИ ВАЖНО: Проверить связанные функции**
   - Перед изменением любого обработчика/функции найти все места, где она используется
   - Проверить, какие другие функции могут быть затронуты
   - Убедиться, что изменения не сломают существующую функциональность
   - Использовать `grep` или `codebase_search` для поиска всех использований

### После выполнения задачи
1. Пройтись по чеклисту из спека
2. **Проверить все связанные функции:**
   - Убедиться, что все обработчики кнопок/команд работают
   - Проверить, что меню отображаются корректно
   - Убедиться, что навигация между меню работает
   - Проверить, что не сломались существующие функции
3. Провести аудит кода:
   - Потенциальные проблемы и уязвимости
   - Неиспользуемые функции
   - Несоответствия patterns.md
   - Проблемы производительности
   - Проблемы безопасности

4. Если найдены проблемы - исправить и обновить документацию

### Правила тестирования перед коммитом
**ОБЯЗАТЕЛЬНО перед каждым коммитом:**
1. Проверить все обработчики, связанные с изменяемым кодом
2. Убедиться, что все кнопки в меню имеют обработчики
3. Проверить, что навигация (кнопки "Назад", "Главное меню") работает
4. Убедиться, что не сломались существующие функции
5. Проверить логику работы с базой данных (если изменялась)
6. Убедиться, что все импорты корректны

**Если что-то сломалось:**
- Немедленно исправить
- Не коммитить сломанный код
- Проверить все связанные функции после исправления

### Обновление UI при изменении логики
**Важное правило:** При изменении логики работы функции обязательно обновлять соответствующие кнопки и их названия:
- Если старая кнопка больше не работает корректно - заменить её новой с правильной логикой
- Если логика изменилась - обновить название кнопки, чтобы оно отражало новое поведение
- Если добавилась новая функциональность - добавить новые кнопки, не удаляя старые без необходимости
- **Пример:** Если кнопка "Удалить старые чеки" не работает, заменить её на "Удалить старые чеки (30+ дней)" и добавить "Удалить все чеки" для полного контроля

### Аудит готового кода
После больших задач можно открыть новый чат и попросить агента провести аудит:
- Найти потенциальные проблемы и уязвимости
- Найти неиспользуемые функции
- Проверить соответствие правилам из базы знаний
- Выявить проблемы производительности

На основе аудита можно составить новое техзадание и отправить агента исправлять проблемы.
